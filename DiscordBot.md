# DiscordBot.md — Life Reroll Discord Bot 구현 스펙

> life-reroll.com의 Discord 봇 확장. 이 문서는 구현 기준 문서다.
> 단계별로 진행하며, 각 단계는 별도 지시로 착수한다. **이 문서를 읽었다고 모든 단계를 한 번에 구현하지 말 것.**

---

## A. 아키텍처 결정 (변경 불가)

1. **봇은 독립 프로세스다.** Next.js 앱에 통합 금지 — API 라우트, 서버 액션, 서버리스 환경에서 봇을 실행하려는 시도 일절 금지. 봇은 Discord 게이트웨이에 상시 연결되는 장기 실행 프로세스이며 별도 배포(Railway/Fly.io)된다.
2. **언어는 TypeScript, 라이브러리는 discord.js.** 기존 웹 코드와 로직 공유를 위해서다.
3. **모노레포 구조:**
   ```
   packages/core   # 국가 데이터, 생 샘플링, 희귀도, 배틀 판정 — 플랫폼 무관 순수 로직
   apps/web        # 기존 Next.js (기존 기능 회귀 금지)
   apps/bot        # discord.js 봇 (core를 import)
   ```
   core는 Discord/Next.js 어느 쪽 의존성도 갖지 않는다. 추후 LINE 어댑터가 3번째 소비자가 된다.
4. **모든 상태는 DB(Postgres/Supabase), 메모리는 캐시로만.** 봇 재시작 시 유실되는 진행 상태가 없어야 한다.
5. **동시성은 원자적 쿼리로 처리한다:**
   - 출생 번호: DB SEQUENCE로만 발급. 앱 레벨 max+1 계산 금지
   - 공덕 차감 등: 조건부 `UPDATE ... WHERE` + 반환 행 수로 성공 판정
6. **버튼/컴포넌트는 stateless.** custom_id에 상태 인코딩(예: `reroll:confirm:{userId}`), 봇 재시작 후에도 동작해야 한다.
7. **슬래시 커맨드 전용.** Message Content Intent 사용 금지. 개발 중 커맨드 sync는 길드 스코프로.
8. **밸런스 수치는 전부 `packages/core/src/config.ts` 한 파일로 분리한다.** 샘플링 σ, 희귀도 합성 공식 계수, 배틀 보정 폭, 공덕 수치 등. **미확정 수치는 임의로 확정하지 말고 합리적 placeholder + `// TODO(balance)` 주석으로 표시할 것.**

## B. 게임 컨셉

- 뽑기 단위는 **생(生)**. 국가는 카드가 아니라 확률 분포 — 국가 확정(실제 인구 비율 가중) 후 그 나라 통계에서 개인 스탯을 샘플링한다.
- 모든 생은 조합상 유일. 출생 번호는 전역 시퀀스.
- 희귀도 2층: 국가 희귀도(뽑힐 확률) × 생 희귀도(분포 내 극단성).
- 경쟁 축은 소유·강탈이 아니라 기록(최장수·최고소득·최희귀).
- 유저의 생 컬렉션 = **덱**. 뽑은 모든 생이 배틀 출전 가능한 로스터가 된다.

## C. 커맨드 스펙

| 커맨드 | 기능 | defer | 출력 |
|---|---|---|---|
| `/환생` | 생 뽑기 (1일 3회, 공덕으로 추가). 직전 생 특성 1개 이월(업 계승) 선택 버튼 | ✅ (LLM) | 공개 |
| `/여권 [생번호]` | 생 상세: 스탯·전적·이름. 생략 시 최신 생 | ❌ | 공개(ephemeral 옵션) |
| `/덱` | 내 로스터 + 최고 기록 3종 하이라이트 | ❌ | ephemeral 기본 |
| `/명명 <생번호> <이름>` | 생에 이름 붙이기 | ❌ | 공개 |
| `/도감` | 서버 공동 국가 컬렉션 n/195. 대륙별 요약 + 페이지네이션 버튼 | ❌ | 공개 |
| `/배틀 @유저` | 자동 판정 배틀 (§E) | ✅ (중계) | 공개 |

- **공개 출력 원칙**: 뽑기·배틀·신규 발견은 절대 ephemeral로 만들지 않는다 (서버 내 목격이 확산 엔진).
- 결과 임베드에 봇 초대 링크 넣지 않는다. footer에 봇 이름만.
- defer 후 15분 내 미응답 시 인터랙션이 죽으므로, LLM 경유 커맨드는 타임아웃(예: 8초) → 폴백 템플릿 경로 필수.

## D. 생 샘플링 (packages/core)

| 스탯 | 소스 | 분포 |
|---|---|---|
| 국가 | 실제 인구 비율 가중 랜덤 | — |
| 수명 | 국가 기대수명 | 정규(σ는 config, TODO(balance)), 평균 대비 편차 표기 |
| 소득 | 국가 1인당 GDP | 로그정규(우측 꼬리), "국가 중위의 n배" 표기 |
| 성별 | 국가 출생 성비 | — |
| 형제 수 | 국가 출산율 | 포아송 |
| 도시/농촌 | 국가 도시화율 | 베르누이 |
| 직업군 | 소득수준별 풀 (정적 테이블) | 인생 요약 입력용 |
| 특성 태그 | 위 조합에서 파생 (「장수」「대가족」 등) | 업 계승 대상 |

- **생 희귀도 점수**: 축별 국가 내 백분위 합성 × 국가 확률. 합성 공식은 config에 두고 TODO(balance). 표기는 "상위 n%".
- 출생 번호 라운드 넘버(#100, #1000, #10000)에 칭호 부여.
- 기존 웹의 국가 데이터셋·확률 로직을 core로 이동해 웹과 봇이 동일 소스를 쓴다.

## E. 배틀 (MVP: 자동 판정)

- 랜덤 3축 3판 2선승. 축 풀과 승리 조건:
  - 수명: 높은 쪽 승 / 소득: 높은 쪽 승 / 형제 수: 많은 쪽 승 / 모국 인구: 많은 쪽 승 / 희귀도: 낮은 확률 쪽 승
- 출전 생: 옵션으로 생번호 지정, 미지정 시 덱에서 축별 최적 자동 선발.
- **비동기**: 상대 수락 불필요 — 상대 덱과 즉시 자동 대결. 같은 상대 1일 3회 상한.
- 각 축 판정에 ±10% 이내 랜덤 보정 (config). 업셋 연출은 축 추첨 결과로 서사화.
- **언더독 보상**: 축별 사전 승률로 종합 기대 승률 계산 → 열세 측 승리 시 공덕 대폭, 우세 측 승리 시 소액 (계수는 config, TODO(balance)).
- **방문 도장**: 승리 시 상대 생의 국가에 도장 → 개인 도감 우회 수집로.
- 생별 wins/losses 누적 (전적이 애착 장치다).
- 결과: 축별 판정은 즉시 계산, 중계 텍스트 1~2문장은 패턴(역전승/순당한 승리/접전)별 사전 템플릿에서 선택. LLM 실시간 중계는 후속 단계.

### 톤 가이드 (모든 유저 노출 문구에 적용)
- 죽음·빈곤을 조롱·희화화하지 않는다. 모든 생은 완결된 삶으로 존엄하게 서술.
- 저소득 국가 생을 "꽝"으로 표현하는 문구 금지. 모든 생은 덱 자산이다.

## G. DB 스키마

```sql
users        (discord_id PK, merit int, created_at)
lives        (id PK DEFAULT nextval('birth_seq'),  -- 출생 번호
              user_id FK, country_code, gender,
              lifespan numeric, income_mult numeric, siblings int,
              occupation, urban bool, traits text[],
              rarity_score numeric, name text NULL,   -- 유저 명명
              wins int DEFAULT 0, losses int DEFAULT 0, created_at)
guilds       (guild_id PK, settings jsonb)
guild_dex    (guild_id, country_code, first_life_id FK, PK(guild_id, country_code))
stamps       (user_id, country_code, PK(user_id, country_code))  -- 방문 도장
battles      (id PK, life_a FK, life_b FK, axes text[], winner FK,
              upset bool, created_at)
hall_of_fame (guild_id, category, life_id FK, PK(guild_id, category))
```

- 일일 뽑기 횟수는 lives의 created_at 카운트로 판정 (별도 카운터 테이블 금지 — 정합성).

## H. 단계별 구현 계획

각 단계는 별도 지시로 착수하며, 단계 완료 시 리뷰를 받는다.

- **1단계 — 모노레포 재구성 + core 추출.** 기존 웹의 국가 데이터·확률 로직을 packages/core로 이동, 웹이 core를 import하도록 수정. **기존 웹 기능 회귀 0이 완료 조건.** config.ts 골격 생성.
- **2단계 — 봇 뼈대 + `/환생`.** discord.js 봇 프로세스, DB 마이그레이션, 생 샘플링 연결, defer·템플릿 폴백, 업 계승 버튼. §A 제약 준수 리뷰 포함.
- **3단계 — `/덱` `/여권` `/명명` `/도감`.** 페이지네이션 포함.
- **4단계 — `/배틀`.** 판정·언더독 보상·방문 도장·중계 템플릿.
- **5단계(별도 판단) — LLM 실연동, top.gg webhook.**
